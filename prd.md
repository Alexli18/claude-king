PRD: Claude King (v1.0) — Оркестратор ИИ-Королевств1. Обзор продукта (Vision)Claude King — это автономная программная оболочка (Host Orchestrator) для профессиональных пользователей ИИ-агентов. Система превращает разрозненные терминалы и процессы в единую иерархическую экосистему («Королевство»), где центральный агент (Король) координирует работу специализированных агентов (Вассалов), обладая полным контекстом состояния системы, файлов и оборудования.2. Целевая аудитория (Target Audience)Power Users ИИ: Разработчики, использующие Claude Code, Aider, Open Interpreter.Инженеры смежных областей: Те, кто одновременно ведет проекты по Hardware (ESP32/UART), ML (инференс/обучение) и Web (Fullstack).Многозадачные профи: Пользователи, ведущие 10–50 изолированных проектов одновременно.3. Ключевые проблемы и решенияПроблемаРешение в Claude KingКонтекстная слепота (Клод не знает, что в соседнем окне упал билд).Global Awareness: Даэмон транслирует события изо всех окон в активную сессию.Шум в логах (LLM захлебывается в мусоре из UART/Debug).RTK (Real-Time Kernel): Семантическая фильтрация и сжатие логов перед подачей в ИИ.Сложность управления (нужно прыгать между 20 окнами tmux).Command Throne: Управление всем проектом напрямую из чата Claude Code через MCP-инструменты.Конфликт контекстов (каша в голове ИИ при работе с разными проектами).Local Sovereignty: Изоляция «Короля» на уровне папки проекта (.king).4. Функциональные требования (Core Features)4.1. Архитектура "Host-Vassal"King Daemon (Go): Фоновый процесс, управляющий PTY-сессиями (Tmux-like), сокетами и MCP-сервером.VMP Protocol: Формат vassal.json в репозиториях для самодекларации скиллов (например, flash_esp32) и артефактов (например, firmware.bin).Artifact Ledger: Единый реестр «субпродуктов», позволяющий передавать файлы между агентами по коротким ссылкам (king://artifacts/...).4.2. Интеграция "Command Throne"Scepter Tools (MCP): Набор инструментов, впрыскиваемых в Claude Code:list_vassals() — статус всех окон.exec_in(target, cmd) — запуск команды в фоновом терминале.read_neighbor(path) — доступ к файлам соседних проектов.Sidecar CLI (kingctl): Утилита внутри терминала для быстрой связи с Даэмоном без использования LLM.4.3. Интеллектуальный слой (RTK)Semantic Sieve: Алгоритм, который превращает поток текста в цепочку событий (Events).Context Injection: Автоматическая подмешка краткой сводки состояния системы (System Health) в каждый промпт Клода.5. Технические требования (Technical Stack)Язык разработки: Go (производительность, работа с PTY, бинарная дистрибуция).Inter-Process Communication: Unix Domain Sockets для мгновенной связи между King и инструментами.Хранение: SQLite (для логов и состояния) + Vector Store (для долгой памяти проекта).Интерфейс: TUI на базе библиотеки bubbletea (интерактивные таблицы, графики нагрузки).6. Пользовательские сценарии (User Stories)Сценарий 1: "Hardware-Software Sync"Я правлю API на Go в Claude Code. В это время в соседнем (скрытом) терминале крутится лог с ESP32. Как только железка выдает ошибку десериализации JSON, Клод в моем основном окне говорит: "Бро, ESP32 ругается на формат даты в твоем последнем коммите. Исправить?"Сценарий 2: "The Grand Overseer"У меня 50 проектов. Я захожу в папку project-x, пишу king up. Даэмон поднимает именно то окружение, те порты и тех вассалов, которые прописаны в этой папке. Клод сразу "вспоминает" контекст именно этого королевства.7. Дорожная карта (Roadmap)Phase 1 (MVP): Бинарник king, создающий PTY-сессию и транслирующий её в базовый MCP-сервер.Phase 2 (Command): Реализация exec_in и проброс инструментов из vassal.json.Phase 3 (Senses): Внедрение RTK-фильтрации и системы алертов между окнами.Phase 4 (TUI): Финальная полировка визуального дашборда.8. Метрики успехаZero-Switching: Пользователь не переключает окна терминала вручную более 90% времени.Context Density: ИИ получает в 10 раз меньше "мусорных" токенов из логов при сохранении точности диагностики.Deployment Speed: Развертывание нового сложного окружения (5+ терминалов) одной командой king up.
