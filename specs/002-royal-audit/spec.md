# Feature Specification: The Royal Audit

**Feature Branch**: `002-royal-audit`
**Created**: 2026-04-02
**Status**: Draft
**Input**: User description: "The Royal Audit — инструментальный слой прозрачности для Claude King с тремя слоями видимости (Ingestion, Sieve, Reasoning), TUI Audit Hall, Time-Travel Debugging и Sovereign Approval."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Real-Time Audit Stream (Priority: P1)

Пользователь запускает Claude King с несколькими вассалами и хочет видеть, что именно система «видит» из терминалов и как данные трансформируются через фильтры. Он открывает TUI-дашборд, переключается на таб "Audit Hall" и видит прокручиваемую ленту записей аудита с тремя слоями: сырой поток (Ingestion), результат фильтрации Sieve, и итоговые события/действия. Пользователь может прокручивать ленту, видеть записи в реальном времени и понять, почему определённая строка была отфильтрована или помечена как Event.

**Why this priority**: Это ядро фичи — без потока аудита остальные возможности (Time-Travel, Sovereign Approval) не имеют смысла. Минимально жизнеспособная ценность: прозрачность обработки данных.

**Independent Test**: Запустить daemon с одним вассалом, выполнить команду генерирующую ошибку, и проверить что в Audit Hall видно: (a) сырую строку Ingestion, (b) решение Sieve (matched/suppressed/no-match), (c) итоговый Event.

**Acceptance Scenarios**:

1. **Given** daemon запущен с вассалом "shell", **When** пользователь открывает TUI и переключается на таб Audit Hall, **Then** он видит ленту записей аудита с колонками: Timestamp, Layer, Source, Content
2. **Given** вассал генерирует 100 строк вывода из которых 1 содержит "ERROR", **When** записи отображаются в Audit Hall, **Then** видно записи слоя Ingestion для каждой строки, 1 запись слоя Sieve (matched), и 1 запись Event
3. **Given** Sieve отфильтровал строку по cooldown-дедупликации, **When** пользователь смотрит Audit Hall, **Then** запись Sieve-слоя показывает причину "suppressed: cooldown"

---

### User Story 2 - Action Trace (Priority: P2)

Пользователь хочет понять, почему AI-агент (Claude) выполнил определённую команду. Каждый вызов exec_in помечается уникальным Trace ID. Пользователь находит команду в Audit Hall и видит полную цепочку: что вызвало команду (Trigger — событие/паттерн), какие данные были доступны агенту в этот момент (Context), и результат выполнения (Execution).

**Why this priority**: Action Trace необходим для отладки поведения AI-агента и является ключевой частью "Explainability by Design". Без него пользователь видит ЧТО произошло, но не ПОЧЕМУ.

**Independent Test**: Выполнить exec_in через MCP, затем найти запись с Trace ID в аудите и проверить наличие полей Trigger, Context, Execution.

**Acceptance Scenarios**:

1. **Given** AI-агент вызывает exec_in для вассала "shell" с командой "npm test", **When** пользователь открывает Action Trace для этой команды, **Then** он видит: Trace ID, команду, имя вассала, время начала, время окончания, exit code, и вывод
2. **Given** exec_in вызван после события типа "error" от Sieve, **When** пользователь смотрит Trigger в Action Trace, **Then** он видит ссылку на исходное событие Sieve
3. **Given** exec_in завершился с ошибкой (exit code != 0), **When** пользователь смотрит Action Trace, **Then** статус записи помечен как "failed" с полным выводом ошибки

---

### User Story 3 - Time-Travel Debugging (Priority: P3)

Пользователь заметил, что система начала вести себя странно 5 минут назад. Он открывает Audit Hall и фильтрует записи по временному диапазону. Он видит «слепок» системы на тот момент: какие вассалы были активны, какие события летели, какие команды выполнялись. Он может передать AI-агенту конкретный временной диапазон для анализа.

**Why this priority**: Time-Travel расширяет базовую функциональность аудита до инструмента ретроспективной отладки. Критично для сценариев с большим количеством проектов, но зависит от рабочего P1.

**Independent Test**: Создать серию событий, затем запросить аудит за определённый временной диапазон через CLI или MCP и проверить, что возвращаются только записи из указанного диапазона.

**Acceptance Scenarios**:

1. **Given** в течение 10 минут произошло 50 аудит-записей, **When** пользователь фильтрует по диапазону "последние 2 минуты", **Then** отображаются только записи за указанный период
2. **Given** пользователь нашёл аномалию в 14:05, **When** он запрашивает снимок системы на 14:05, **Then** он видит: активные вассалы, последние события, последние exec_in команды на тот момент
3. **Given** аудит-записи хранятся в хранилище, **When** пользователь запрашивает записи старше retention-периода, **Then** система сообщает что данные за этот период удалены

---

### User Story 4 - Sovereign Approval (Priority: P4)

Пользователь включает режим `sovereign_approval` в конфигурации kingdom.yml. Теперь перед каждой командой exec_in система приостанавливается и выводит запрос подтверждения в Audit Hall и TUI. Пользователь видит: что собирается сделать агент, почему, и может одобрить или отклонить.

**Why this priority**: Sovereign Approval — продвинутый режим контроля. Он важен для безопасности, но является надстройкой над базовым аудитом. Для MVP достаточно P1-P3.

**Independent Test**: Включить sovereign_approval в конфиге, вызвать exec_in через MCP и проверить, что команда НЕ выполняется до получения подтверждения пользователя.

**Acceptance Scenarios**:

1. **Given** sovereign_approval включён в kingdom.yml, **When** AI-агент вызывает exec_in, **Then** команда ставится в очередь ожидания и в Audit Hall появляется запрос подтверждения
2. **Given** запрос подтверждения отображён, **When** пользователь одобряет действие, **Then** команда выполняется и результат записывается в аудит
3. **Given** запрос подтверждения отображён, **When** пользователь отклоняет действие, **Then** команда не выполняется, в аудит записывается "rejected by user", AI-агент получает ответ об отклонении
4. **Given** sovereign_approval выключен, **When** AI-агент вызывает exec_in, **Then** команда выполняется немедленно без запроса подтверждения

---

### Edge Cases

- Что происходит при высокой нагрузке (тысячи строк в секунду от активного вассала)? Ingestion Layer должен иметь механизм throttling/sampling.
- Как система ведёт себя при заполнении хранилища? Retention policy должна автоматически удалять старые записи.
- Что если пользователь фильтрует по несуществующему временному диапазону? Система возвращает пустой результат без ошибки.
- Как обрабатывается Sovereign Approval в headless-режиме (без TUI)? Запросы доступны через CLI с настраиваемым таймаутом.
- Что происходит с pending approval-запросами при перезапуске daemon? Pending запросы помечаются как "expired" при рестарте.
- Что если Ingestion Layer включён, но вассал генерирует бинарные данные? Непечатные символы заменяются placeholder-ом.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система ДОЛЖНА записывать каждую строку вывода вассала в аудит-лог на уровне Ingestion Layer (при включённом режиме ingestion)
- **FR-002**: Система ДОЛЖНА записывать каждое решение Sieve (matched, suppressed, no-match) в аудит-лог на уровне Sieve Layer
- **FR-003**: Система ДОЛЖНА записывать каждый вызов exec_in с уникальным Trace ID, включая входные параметры, результат и время выполнения (Action Layer)
- **FR-004**: Система ДОЛЖНА предоставлять TUI-таб "Audit Hall" с прокручиваемой лентой записей в реальном времени
- **FR-005**: Записи аудита ДОЛЖНЫ содержать: timestamp, layer (ingestion/sieve/action), source (имя вассала), content и metadata
- **FR-006**: Система ДОЛЖНА поддерживать фильтрацию записей по временному диапазону, слою и источнику (вассалу)
- **FR-007**: Система ДОЛЖНА хранить аудит-записи с настраиваемым сроком хранения (retention)
- **FR-008**: Система ДОЛЖНА предоставлять MCP-инструмент `get_audit_log` для программного доступа к записям аудита с фильтрами
- **FR-009**: Система ДОЛЖНА поддерживать режим Sovereign Approval, при котором exec_in команды требуют явного подтверждения пользователя
- **FR-010**: Система ДОЛЖНА предоставлять CLI-команду для просмотра записей аудита с фильтрами
- **FR-011**: Action Trace ДОЛЖЕН включать: Trace ID, trigger (ссылка на вызвавшее событие), контекст, execution (результат команды)
- **FR-012**: Система ДОЛЖНА автоматически удалять аудит-записи старше настроенного retention-периода

### Key Entities

- **AuditEntry**: Одна запись аудита — timestamp, layer (ingestion/sieve/action), source (vassal name), content, trace_id (для action layer), metadata
- **ActionTrace**: Расширенная запись для exec_in — trace_id, trigger_event_id, command, vassal, exit_code, duration, output
- **ApprovalRequest**: Запрос подтверждения — trace_id, command, vassal, reason, status (pending/approved/rejected), responded_at

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Пользователь может просмотреть полную цепочку обработки данных (от сырого вывода до итогового события) за менее чем 10 секунд навигации
- **SC-002**: Все exec_in команды имеют трассируемую историю с Trace ID, доступную через TUI и CLI
- **SC-003**: Фильтрация аудита по временному диапазону возвращает результаты за менее чем 2 секунды на объёме до 100 000 записей
- **SC-004**: В режиме Sovereign Approval ни одна команда exec_in не выполняется без явного подтверждения пользователя
- **SC-005**: Аудит-лог управляется retention-политикой и автоматически очищается без ручного вмешательства
- **SC-006**: Пользователь может ретроспективно найти причину аномалии, используя Time-Travel фильтрацию, без обращения к внешним инструментам

## Assumptions

- Feature 001 (AI Kingdom Orchestrator) полностью реализована и стабильна: daemon, PTY manager, Sieve, Store, MCP server
- Для MVP используется SQLite как единственное хранилище аудит-записей (без LanceDB)
- Ingestion Layer (запись каждой строки) включается отдельной настройкой, так как создаёт значительную нагрузку на хранилище; Sieve и Action слои записываются всегда
- Sovereign Approval работает через TUI (интерактивно) и через CLI (с таймаутом для headless-режима)
- Retention по умолчанию — 7 дней для всех слоёв, настраивается в kingdom.yml
- MCP-инструмент `get_audit_log` доступен AI-агентам, но Sovereign Approval ответы доставляются пользователю отдельно (через TUI/CLI, не через MCP)
